# app/macos

The macOS frontend for LoFi. Swift + AppKit on top of the shared Rust core (`app/core/`) via a C ABI, built by Bazel.

## Status

Builds and runs on macOS 26 Tahoe with Xcode 26. `bazelisk run //app/macos:launch` floats a borderless panel listing every `.app` under `/System/Applications`, `/Applications`, and `~/Applications`. The panel renders as a rounded, translucent Liquid Glass surface with a drop shadow ‚Äî Spotlight's look (a vibrant `NSVisualEffectView` is the fallback below macOS 26, since the deployment target is 15.0). It has a focused search field at the top that fuzzy-filters as the user types ‚Äî borderless, with its magnifier aligned to the list's icon column ‚Äî and each row renders as `[icon] Name ‚Ä¶ [Category]` with the category dimmed and trailing-aligned. Application rows whose owning bundle had at least one open window at summon time also get a small **running-indicator dot** drawn under the icon ‚Äî a 6pt `secondaryLabelColor` circle, mirroring the GNOME launcher's `.running-indicator` (see `app/gnome/src/ui.rs`). The "is this app running?" check is a one-pass scan of `WindowDiscovery.discover()`'s freshly-gathered bundle ids per summon (no observers, no caching), so the dot reflects the state at the moment Option+Space was pressed. The dot view (`EntryRowView.RunningDotView`) is always part of the icon column at its full 6√ó6 size ‚Äî only its layer's `backgroundColor` toggles between `secondaryLabelColor` and clear ‚Äî so row heights don't shift between running and not-running entries. Activation history persists across launches via the SQLite store at `~/Library/Application Support/dev.jplein.lofi/mru.sqlite`, so the apps you actually use bubble to the top on every subsequent launch. With Screen Recording **and** Accessibility permissions granted, the **window-action commands** appear as `"Command"` rows whenever there is any non-LoFi window open on the active Space. They act on the **globally frontmost** non-LoFi window via the AX API (GNOME parity ‚Äî the window the user was just using, regardless of which display it's on). There are fourteen GNOME-parity commands available on every macOS configuration ‚Äî Center, Center third, Center half, Center two-thirds, Left third, Left half, Left two-thirds, Right third, Right half, Right two-thirds, Standard size, Minimize, Toggle maximize, Toggle fullscreen (move/resize for the eleven geometry kinds, minimize/fullscreen/maximize toggles for the three state kinds) ‚Äî plus two macOS-only multi-display commands appended when 2+ displays are attached: **Next display** and **Previous display**. Within each of the center / left / right groups the fractions ascend (third, half, two-thirds); the right-aligned variants stay flush against the work area's right edge because their geometry anchors on `x + width - w` rather than `x + w` (so integer-division rounding never leaves a gap). The multi-display commands move the target window to the next/previous display (NSScreen.screens order, with wrap-around), preserving the window's offset from the work-area origin and clamping size/position to the destination display's work area. The geometry commands act on the target window's own display (so "Left half" of a window on display 2 stays on display 2). No non-LoFi window (or missing permissions) ‚áí no command rows. See *Permissions* below for how the two TCC grants interact. The **power commands** ‚Äî Lock, Log Out, Sleep, Restart, Shutdown ‚Äî appear as `"PowerCommand"` rows unconditionally (no window, no permissions required); they dispatch through `PowerCommands.swift` (CGSession / pmset / direct Apple events to loginwindow). See *Power commands* below for the dispatch routing and why we go through `loginwindow` rather than `System Events`.

**The window switcher (listing open windows as launcher entries) is intentionally disabled on macOS.** It was implemented and shipped, then disabled after empirical confirmation that two macOS Tahoe limitations make it unreliable for an unprivileged-process launcher: cross-Space activation is impossible without Dock injection (gotcha 13), and AX writes that retarget another app's key window across a display boundary are silently dropped (gotcha 14). The scoping workaround (active Space + active display) still leaves enough holes ‚Äî the list contents would depend on the mouse cursor location, AND picking a listed same-display window can still fail to focus if the app has sibling windows on other displays ‚Äî that we'd rather not ship the feature than ship one that's unpredictable. See *Out of scope* below and the *gotchas* section for the full investigation.

LoFi runs as a long-running background daemon: a single `bazelisk run :launch` starts it, after which it listens for the **Option+Space** global hotkey (and Launch Services reopen events from `:activate`). Each press of the hotkey gathers the current frontmost-non-LoFi window, rebuilds the entry list, applies MRU, and shows the panel; pressing Esc, clicking outside, or activating an entry hides the panel and returns LoFi to the background. Cmd-Q (from the hidden Application menu) is the explicit quit path; `bazelisk run :close` quits an existing instance from the terminal.

## Why a separate frontend

The cross-platform core (`lofi-core`) holds the data model and pure logic ‚Äî `Application`, `Entry`, `EntryRef`, fuzzy matcher, MRU store. Anything that depends on a particular window system or app-discovery mechanism lives in the platform crate. On GNOME that's `app/gnome/` (GTK4 + libadwaita + D-Bus to the Shell extension); on macOS that's this directory (AppKit linking the `liblofi_core.a` produced by `rules_rust`, with `lofi_core.h` exposed as a Clang module via `rules_swift`).

The two frontends share nothing at the windowing-system level, so they're separate projects. Sharing the data model and logic (in `core/`) is what keeps fuzzy match, MRU, and command activation behaviour consistent between platforms.

## Why Swift drives discovery and Rust holds the list

Same pattern as `app/gnome/`: the platform layer is the gatherer, the core is the canonical store.

- `AppDiscovery.discover()` walks `/System/Applications`, `/Applications`, and `~/Applications`, dedups by bundle identifier (first-wins, in that root order so Apple's stock apps shadow any same-bundle-id third-party installs), and returns a sorted list. Mirrors GNOME's `app/gnome/src/apps.rs` first-dir-wins policy.
- `AppDelegate` pushes each discovered `.app` into the Rust-owned `EntryList` via `lofi_entries_push_application(...)`. After that point the list belongs to Rust; Swift reads it back through `lofi_entries_len` and the `lofi_entries_get_*` accessors (name, bundle id, category, icon, window id, running flag, command id/geometry, power-command id). See `app/core/README.md` for the authoritative FFI surface.

This shape leaves the matcher, MRU, and future activation logic on the Rust side without having to expose `Application`/`Entry` as Swift types. Adding an `EntryRef`-based MRU lookup in a future slice is a Rust change, not a Swift one.

## Why Bazel

Bazel owns the macOS build graph end to end. `rules_rust` + `crate_universe` consume `app/Cargo.lock` and produce `liblofi_core.a`; a `genrule` runs the (Bazel-built) `cbindgen` binary to emit `lofi_core.h`; `cc_library` + `swift_interop_hint` expose the header to Swift as the `LoFiCore` Clang module; `rules_swift` compiles the `.swift` sources; `rules_apple`'s `macos_application` packages everything into `LoFi.app`.

The earlier xcodegen + xcodebuild + cargo + bash-script pipeline is gone ‚Äî one front door (Bazel), one build graph, one set of incrementality rules. The Linux GNOME crate still goes through Cargo + Crane in `flake.nix`; Bazel is macOS-only for now. Both paths read the same `Cargo.lock` so dependency versions stay coherent.

## Build / run

```sh
bazelisk build //app/macos:LoFi       # produce bazel-bin/app/macos/LoFi.zip
bazelisk run   //app/macos:launch     # quit any running instance, unzip + `open` the freshly-built bundle
bazelisk run   //app/macos:close      # quit any running instance (idempotent)
bazelisk run   //app/macos:activate   # send a re-open event to the running instance (summons the panel); no-op otherwise
bazelisk run   //app/macos:install    # install to ~/Applications + register a LaunchAgent so LoFi starts at login
bazelisk test  //app/core:ffi_test    # run the 52 FFI integration tests
bazelisk run   //app/macos:xcodeproj  # regenerate app/macos/LoFi.xcodeproj
```

`:launch` is the dev-cycle target ‚Äî every invocation guarantees you are interacting with the binary Bazel just built, not a stale background process. `:close` is the idempotent shutdown. `:activate` is the production way to summon the panel ‚Äî it sends Launch Services a re-open event that `AppDelegate.applicationShouldHandleReopen` translates into a `summonPanel` call. `:install` is the production setup path: it writes `LoFi.app` to `~/Applications` and a per-user LaunchAgent plist to `~/Library/LaunchAgents/dev.jplein.lofi.plist`, then `launchctl bootstrap`s the agent so the daemon starts immediately and resumes at next login. All four quit-paths use a graceful AppleScript quit first and SIGTERM as a fallback, so persistent stores (`mru.sqlite`, `SavedFrameStore`) get to flush.

The LaunchAgent uses `KeepAlive = { SuccessfulExit = false }` ‚Äî launchd restarts LoFi if it crashes (non-zero exit), but Cmd-Q / `:close` quits cleanly and stays quit until next login. To re-launch a stopped-but-installed daemon mid-session: `launchctl kickstart gui/$UID/dev.jplein.lofi`. To uninstall: `launchctl bootout gui/$UID/dev.jplein.lofi && rm -rf ~/Applications/LoFi.app ~/Library/LaunchAgents/dev.jplein.lofi.plist` (no dedicated `:uninstall` target ‚Äî that's a one-liner).

Always invoke the build through `bazelisk`, not `bazel`. The Nix devShell installs `bazelisk` (a thin launcher that reads `.bazelversion` and downloads the pinned Bazel release on demand); it does *not* install a `bazel` binary directly. Calling `bazel` outside the devShell will either fail with "command not found" or pick up a system Bazel that doesn't match `.bazelversion`.

`bazelisk run //app/macos:LoFi` also "works" ‚Äî it invokes rules_apple's stock launcher script, which extracts the bundle to `/tmp` and execs the binary directly. The downside is it bypasses Launch Services, so `LSUIElement=YES` activation can behave subtly differently. `:launch` routes through `open`, which matches the production launch path.

For Xcode-based debugging, run `bazelisk run //app/macos:xcodeproj` and open `app/macos/LoFi.xcodeproj`. The project is generated by `rules_xcodeproj` from the Bazel graph ‚Äî Bazel still drives the actual compile; Xcode is the IDE and debugger surface only. The `.xcodeproj` is gitignored; regenerate after Bazel-graph changes.

First-time build downloads Bazel (per `.bazelversion`), then the rule stacks, then resolves the Cargo lockfile via `crate_universe`. Subsequent builds hit the action cache and finish in seconds.

`DEVELOPER_DIR` must point at the user's Xcode install (`/Applications/Xcode.app/Contents/Developer`) before invoking `bazelisk` ‚Äî `.envrc` does this. Without that override the Nix devShell leaves `DEVELOPER_DIR` pointing at a partial nix-store SDK and `rules_swift` bails with "Could not determine Xcode version at all."

## Formatting / linting

Swift is formatted and linted with Apple's `swift-format`, which ships *inside* the Xcode toolchain (`xcrun swift-format`) ‚Äî there is no separate tool to install or version-pin. That mirrors the reasoning for running the Rust gates through the rules_rust toolchain rather than a parallel cargo install (see `app/core/README.md`): reuse the toolchain the build already depends on. swift-format is both the formatter and the linter, so one script covers both roles:

```sh
./check.sh         # lint: check-only, non-zero exit on any deviation (the gate)
./check.sh --fix   # reformat in place
```

The config is `app/macos/.swift-format` ‚Äî swift-format's defaults with one change: 4-space `indentation` (the existing code style; swift-format defaults to 2). This is deliberately *not* wired into `bazel test`: `rules_swift` has no swift-format aspect (unlike the rules_rust clippy/rustfmt aspects the Rust gates use), so the Swift check runs standalone. swift-format is a formatter plus a *light* linter ‚Äî its default rules don't cover deeper lint concerns (it won't flag, e.g., force-casts), so it complements manual review rather than replacing it.

## Permissions

Window-action commands depend on two separate macOS TCC (Transparency, Consent, and Control) grants. Apps (the `.app` enumeration path) need neither ‚Äî `AppDiscovery` reads bundle directories that are world-readable, and `NSWorkspace.open` does not require special entitlements. The launcher therefore degrades gracefully: if either permission is denied, the panel still lists every installed app and Enter still launches it. The Command rows are the only thing that disappears. (The window *switcher* ‚Äî listing open windows ‚Äî is unconditionally disabled on macOS regardless of permissions; see gotchas 13-14.)

Why two permissions rather than one: macOS treats *reading* on-window state (titles, geometry) and *driving* other processes (raising a specific window) as distinct privacy surfaces, gated by different TCC categories.

- **Screen Recording** (`NSScreenCaptureUsageDescription` in `Info.plist`) ‚Äî required for `CGWindowListCopyWindowInfo` to return `kCGWindowName` strings. The window-action command target picker reads titles via this API to filter out titleless windows (system UI, menu bars, etc.) so the frontmost-non-LoFi pick is reliable. Without it the API still returns one entry per on-screen window, but with empty titles, and `WindowCommands.gatherTarget` then finds no usable target ‚Üí no command rows.
- **Accessibility** (no Info.plist key; `AXIsProcessTrustedWithOptions` is the API contract) ‚Äî required for `AXUIElementPerformAction` (raise) and `AXUIElementSetAttributeValue` (move/resize/state) on another process's windows. Without it the AX calls return `kAXErrorAPIDisabled` and the commands are impossible. Symmetric to Screen Recording: no command target ‚Üí no command rows.

To grant: System Settings ‚Üí Privacy & Security ‚Üí Screen Recording (toggle LoFi on) and Privacy & Security ‚Üí Accessibility (toggle LoFi on). The first Option+Space summon that finds either permission missing fires both system prompts (`CGRequestScreenCaptureAccess` and `AXIsProcessTrustedWithOptions(prompt: true)`) and **returns without showing the launcher panel**. The reason: LoFi's panel runs at `.floating` window level, which composites *above* the centered TCC dialogs ‚Äî and window level beats activation for z-order, so `NSApp.activate(ignoringOtherApps:)` cannot push our panel behind the dialog. Showing the panel would therefore bury the very dialogs the user needs to act on. The prompt fires once per process (`AppDelegate.promptedForPermission`); later summons in the same process show the panel in degraded apps-only mode, and after the user grants + relaunches the new process never reaches this path. Subsequent launches won't re-prompt because TCC remembers the user's decision.

**Relaunch to pick up newly-granted permissions.** This is the most surprising part of the UX: `CGPreflightScreenCaptureAccess` and `AXIsProcessTrustedWithOptions` capture TCC state at process start, so granting a permission while LoFi is running has no effect until the next launch. This is Apple's design ‚Äî TCC permissions are baked into the process's sandbox context at exec time so a long-running daemon can't be silently granted new capabilities mid-session. The practical consequence: the first-launch flow is *launch, grant, relaunch, grant, relaunch* in the worst case where the user grants Screen Recording first, relaunches, then grants Accessibility. The Gotchas section below calls this out again as item 10.

## Power commands

The five power-command rows (Lock, Log Out, Sleep, Restart, Shutdown) mirror GNOME's `power::gather_power_commands` set ‚Äî the same five `PowerCommandKind` variants from `lofi-core`, just dispatched through macOS surfaces instead of D-Bus. They are always present in the launcher (no permission gating, no dependency on a focused window) and dispatch through `PowerCommands.swift`. Three different mechanisms back the five commands:

- **Lock** synthesizes the ‚åÉ‚åòQ keystroke via `CGEvent.post(tap: .cghidEventTap)`. ‚åÉ‚åòQ is the macOS system-wide Lock Screen shortcut (Apple menu ‚Üí Lock Screen invokes the same path). Works with the **Accessibility** permission LoFi already holds for window-action commands ‚Äî synthetic keyboard events posted to the HID event tap are gated by the same TCC category. The older `CGSession -suspend` binary (`/System/Library/CoreServices/Menu Extras/User.menu/Contents/Resources/CGSession`) was removed in macOS Sonoma / Tahoe and has no stable command-line replacement.
- **Sleep** shells out to `/usr/bin/pmset sleepnow`. `pmset sleepnow` accepts the active user without `sudo` (unlike `shutdown`, which requires root). No confirmation dialog ‚Äî matches the GNOME `Suspend` behavior which also skips the polkit prompt.
- **Log Out / Restart / Shutdown** use in-process `NSAppleScript` to send raw Apple events directly to the `loginwindow` process: `¬´event aevtrlgo¬ª` (`kAELogOut`), `¬´event aevtrrst¬ª` (`kAERestart`), `¬´event aevtrsdn¬ª` (`kAEShutDown`). macOS shows its own native confirmation dialog with the 60-second countdown for these three ‚Äî GNOME parity, where the same three commands route through GNOME's `SessionManager` for the same dialog.

Why direct Apple events to `loginwindow` rather than `tell application "System Events" to log out / restart / shut down`: the System Events path triggers an **Automation** TCC prompt (an additional permission on top of Screen Recording + Accessibility) the first time it runs, which is non-obvious user friction. Apple events sent directly to `loginwindow` are exempt from that gate ‚Äî the four-byte event codes are documented since classic Mac OS and `loginwindow` honors them without an Automation grant. We use in-process `NSAppleScript` rather than shelling out to `osascript` for the same reason: a shell-out would make `/usr/bin/osascript` the caller identity for TCC, which would confuse the user if a future macOS version did decide to gate `loginwindow` events. In-process keeps the caller stable as LoFi.

## Gotchas worth calling out

Each cost real time to figure out the first time; each is permanent in the code or build setup with a comment pointing back here. If something in this list starts looking redundant, double-check it really is ‚Äî the cell-rendering bug below was three separate issues stacked, and stripping any one of them brings the blank panel back.

### AppKit

1. **`canBecomeKey` returns `false` for borderless `NSPanel` by default.** Without overriding, the panel renders but never receives keyboard input ‚Äî typing into what looks like a focused launcher just goes to whatever app was previously frontmost. `LoFiPanel: NSPanel` overrides this to `true` (see `PanelController.swift`).
2. **`LSUIElement=YES` launches the process in the background.** No Dock icon, no menu bar ‚Äî but also, the panel never becomes key on its own. `AppDelegate` must call `NSApp.activate(ignoringOtherApps: true)` before showing the panel; without that step the borderless window appears but stays inert.
3. **`LSUIElement=YES` also suppresses the system Application menu, so Cmd-Q has no handler.** `AppDelegate.installHiddenMenu()` installs a minimal `NSMenu` containing only a `Quit` item with `keyEquivalent: "q"`. The menu never becomes visible (still LSUIElement), but its key equivalent fires.
4. **`NSScrollView` does not auto-resize its `documentView`.** A bare `NSTableView()` set as `documentView` sits at 0√ó0 inside the scroll view and never asks for cell views ‚Äî the table is alive (clicks select rows, the scroll wheel "scrolls") but draws nothing. `AppListController` constructs the table with an explicit non-zero `frame` and pairs it with `columnAutoresizingStyle = .uniformColumnAutoresizingStyle`.
5. **`NSTableView.dataSource` and `.delegate` are weak.** If the only strong reference to the list controller is a local variable inside `applicationDidFinishLaunching`, the controller deallocates when that method returns and the table silently stops calling `viewFor:row:` ‚Äî rows scroll and select normally because `numberOfRows` is cached, but cells render blank. `AppDelegate` keeps a strong `listController` property; do not "simplify" it away.
6. **`??` does not fall through empty strings, only `nil`.** Some apps set `CFBundleDisplayName` to `""` rather than omitting the key, which a naive `(displayName as? String) ?? (bundleName as? String) ?? basename` accepts as a valid empty string. `AppDiscovery.discover()` uses a `nonEmpty()` helper to coerce empty-string Info.plist values to `nil` so the fallback chain works.
7. **`panel.initialFirstResponder = searchField` must be set *before* `makeKeyAndOrderFront(_:)`.** Setting it afterwards is a silent no-op ‚Äî by the time the panel orders front it has already picked a default responder, and assigning `initialFirstResponder` later does not retroactively re-route the focus. The user sees the panel appear with the cursor "in" the search field visually but typing goes nowhere. `PanelController.swift`'s init wires this up in the right order; do not move the assignment.
8. **`NSStackView` with `.leading` alignment leaves the search field at its intrinsic narrow width.** A stack view in vertical orientation sizes its arranged subviews to their intrinsic content size on the cross axis, and `NSSearchField`'s intrinsic width is ~100pt ‚Äî far narrower than the panel. The panel pins the search field's leading and trailing anchors to the stack so the field spans the full panel width.
9. **Safari hides in a Cryptex, and the `/Applications/Safari.app` symlink is invisible to bundled apps.** Modern macOS keeps Safari in `/System/Cryptexes/App/System/Applications/Safari.app` (Rapid Security Response can update Safari/WebKit without an OS update). There's a symlink at `/Applications/Safari.app` pointing into the Cryptex, but two things conspire to hide it from a naive enumeration: (a) `FileManager.enumerator(at:options:)` silently skips symlinks-to-directories regardless of options, and (b) the running `LoFi.app` gets a TCC-filtered listing of `/Applications` that omits the Cryptex symlink even when reading via `contentsOfDirectory(at:)`. `AppDiscovery` does a manual recursion (so it would catch *any* symlinks not in the Cryptex case) **and** scans `/System/Cryptexes/App/System/Applications` directly as a fourth root, which is readable from bundled apps and contains Safari (resolved via `/System/Volumes/Preboot/...`).
10. **TCC state for Screen Recording and Accessibility is captured at process start.** `CGPreflightScreenCaptureAccess` and `AXIsProcessTrustedWithOptions` return whatever the state was when LoFi launched, not whatever the state happens to be when we call them. A user who grants either permission while LoFi is already running won't see windows appear in the panel until the next launch ‚Äî the running process is still operating against the TCC context it was forked with. This is Apple's design (a long-running daemon can't be silently granted new privileges mid-session) and is mechanically the same behavior Raycast, Alfred, and CleanShot exhibit. It's still non-obvious for users coming from GNOME where permission checks happen dynamically against the polkit/Wayland session. The Permissions section above documents this for end users; the gotcha here is so we don't "fix" it by, e.g., polling the API or re-checking after `NSApp.activate` ‚Äî the API answer is frozen.
11. **Bridge `CGWindowID` ‚Üí `AXUIElement` by id (`_AXUIElementGetWindow`), with title matching only as a fallback.** We capture each window's `CGWindowID` from `CGWindowListCopyWindowInfo`, but the AX API addresses windows as opaque `AXUIElement`s with no public id accessor. The bridge is the private HIServices function `_AXUIElementGetWindow(element, &cgWindowId)` ‚Äî resolved at runtime via `dlsym(RTLD_DEFAULT, "_AXUIElementGetWindow")` in `WindowControl.swift` (rather than `@_silgen_name`) so that if a future macOS drops the symbol we degrade to title matching instead of failing to launch with an unresolved-symbol error. `AXWindowFinder.match` walks the app's AX windows and returns the one whose `_AXUIElementGetWindow` id equals the one we captured. yabai, Hammerspoon, and AeroSpace all rely on this same symbol.

    This exact-id match is what makes the commands (and `raise`) reliable. The *original* approach matched on title, which **silently broke for apps that retitle**: Ghostty (a terminal) rewrites its window title on every command / cwd change, and browsers retitle on tab switches, so the title captured at gather time no longer matched at activation ‚Äî `find` returned nil and the command no-op'd. The CGWindowID is stable for the window's lifetime, so id matching is immune to that drift.

    Title matching survives only as a fallback for apps whose AX windows don't expose an id, and it still can't use naive `==` because CGWindow and AX titles disagree two ways: (a) CGWindow truncates long titles with a Unicode ellipsis (`‚Ä¶`, U+2026) while AX keeps the full text ‚Äî CGWindow `"Where has Safari gone, and why‚Ä¶"` never `==` the full AX title; (b) AX often appends the owning app's name ‚Äî Chrome: CGWindow `"Hacker News"` vs AX `"Hacker News - Google Chrome"`. `AXWindowFinder.titleMatches` handles both by taking the pre-ellipsis prefix of the CGWindow title and accepting any AX title that *starts with* it. Both `WindowActivation.raise` and the window-action commands go through `AXWindowFinder` (id-first, title-fallback).
12. **Firefox (and some Chromium derivatives) ship with the AX runtime asleep.** `AXUIElementCopyAttributeValue(app, kAXWindowsAttribute, ...)` returns success with an empty array even when the app clearly has windows open ‚Äî calling `kAXRaiseAction` is therefore impossible. The wakeup is `AXUIElementSetAttributeValue(app, "AXEnhancedUserInterface", kCFBooleanTrue)`, the same nudge VoiceOver and Mac scripting tools use; once written, the next `kAXWindowsAttribute` read returns the real list. `AXWindowFinder.windowsForApp` (in `WindowControl.swift`, shared by `WindowActivation.raise` and the command dispatch) does this unconditionally ‚Äî apps that already expose AX just no-op the write. The attribute is undeclared (no `kAX` constant), hence the bare string literal. If the post-kick window list is still empty (sandboxed apps that can't be inspected at all), `raise` falls back to bare `NSRunningApplication.activate()`: the app comes forward and macOS picks one of its windows. We lose precise window targeting, but that's strictly better than dropping the activation. **Flip side:** leaving `AXEnhancedUserInterface` *on* breaks window geometry, so the command path turns it back off before any move/resize ‚Äî see gotcha 20.
13. **Cross-Space activation from a regular macOS process is unworkable on Tahoe ‚Äî one of two reasons the window switcher is disabled (the other is gotcha 14).** The obvious-looking path (the call yabai uses!) leads to a broken Mission Control state and several hours of debugging before you reach the same conclusion.

    `kAXMainAttribute` is a hint the system ignores for Space-switching; `NSRunningApplication.activate()` honors the user's *"When switching to an application, switch to a Space with open windows for the application"* preference (System Settings ‚Üí Desktop & Dock) but doesn't override it. The private SkyLight call `SLSManagedDisplaySetCurrentSpace` ‚Äî the call yabai, Hammerspoon, AeroSpace, and Spaces.app use to force a switch ‚Äî is what we tried.

    On Tahoe (macOS 26) the call **returns `0` (success)** but does not actually commit the visual Space switch when invoked from a regular non-Dock-injected process. Worse, macOS then "reconciles" the inconsistency between the just-changed internal state (current Space = the target) and the unchanged visual state (still rendering the originating Space) by yanking windows from the target Space onto the originating Space ‚Äî a broken Mission Control state (full-size, doesn't resize with the others, recovers only when the user manually switches to the target Space). The yank happens **even with zero downstream work** ‚Äî no AX raise, no `NSRunningApplication.activate()`, no `kAXMain` write, no `AXEnhancedUserInterface` kick, panel hidden, process deactivated, run loop pumped 300ms. The SLS call alone, from our process, is enough.

    The reason is the same one yabai documents extensively: SkyLight gates these Space-switching calls on the *caller's* process identity. yabai, Hammerspoon, and AeroSpace work around this by installing a scripting addition into the Dock process (SIP-disabled) and routing the SLS calls through a UNIX socket ‚Äî calls originating from the Dock are honored. That dock-injection requirement is non-negotiable; we are not implementing it for a launcher.

    Investigated symbols (all resolved successfully via `dlsym(RTLD_DEFAULT, ...)`): `SLSMainConnectionID`, `SLSCopySpacesForWindows` (returns the target Space id correctly), `SLSCopyManagedDisplaySpaces` (returns the per-display Space tree correctly with matching `id64`s), `SLSManagedDisplaySetCurrentSpace` (returns `0` but no visual effect). The investigation lives in this project's memory under `project_sls_cross_space.md`; do not re-attempt without dock injection.

    The first attempted workaround was to scope the window list to the active Space (omit cross-Space windows that we can't activate). That sidestepped the SLS yank but left the cross-display focus problem in gotcha 14, so the window-list feature was disabled entirely. `WindowDiscovery.discover` is still called for the window-action *command* target (the frontmost non-LoFi window on the active display) and for `SavedFrameStore.prune`; the per-window `entries.pushWindow` loop is intentionally absent from `AppDelegate.applicationDidFinishLaunching`. If the user wants cross-Space activation, the macOS-native way is to leave the "switch to a Space with open windows" preference ON and use Cmd-Tab or the Dock.
14. **Cross-display focus retargeting through AX silently fails ‚Äî the other reason the window switcher is disabled.** Sibling to gotcha 13: another thing the OS won't let an unprivileged process do, with several hours of debugging required to confirm.

    Symptom (observed on macOS 26 Tahoe, two displays, "Displays have separate Spaces" enabled):
      - **Finder**, two windows ("Blue Steel HD" on display 1, "Dustbin" on display 2). Picking either: the right window comes visually to the front of its display, the *other* window also raises to the front of its display (`NSRunningApplication.activate()` raises every window of the app to its display's top), and **the previously-focused window stays focused** ‚Äî typically the one on a display other than LoFi's. Even narrowing the list to the active display didn't fix this: picking a same-display window like "Blue Steel HD" still failed to focus it when "Dustbin" was open on the other display, because Finder's key-window state spans displays.
      - **Firefox**, two windows (one per display). Same shape: visual raise correct, focus stays on whichever window was front before LoFi.

    The activation sequence that gets the visual right is: `NSRunningApplication.activate()` ‚Üí `kAXRaiseAction` ‚Üí `kAXMainAttribute = true` (window) ‚Üí `kAXFocusedAttribute = true` (window). All four calls return `.success`. None of them moves the OS-level key-window pointer. We confirmed this with a read-back: after writing `kAXFocusedWindowAttribute` on the AXApplication to point at the target, immediately reading the attribute back returns the *previously-focused* window unchanged. The write is silently dropped.

    Why both `kAXMain` and `kAXFocused`: AppKit apps key on `kAXMain` for "active document window" semantics, while Gecko/Chromium toolkits track their own "current window" independent of macOS z-order and respond to `kAXFocused`. The union covers both conventions. Best-effort ‚Äî a failed set on a missing attribute is the expected shape, not a bug. The order ‚Äî activate first, *then* the per-window writes ‚Äî is what Hammerspoon and Rectangle use; doing it the other way round (writes then activate) is even worse because the writes go to a still-background app that ignores them outright. But neither order recovers focus across a display boundary.

    What does *not* work, and why we shipped feature-disabling instead:
      - **App-level `kAXMainWindowAttribute` / `kAXFocusedWindowAttribute`** on the AXApplication element ‚Äî the most direct retargeting API. Silently dropped (confirmed via read-back).
      - **Synthetic `CGEvent` mouse clicks on the title-bar drag area.** Bypasses the AX gate in principle (real input events), tried it. In practice: the cursor warps visibly (annoying), the LoFi panel blocks clicks on its own display (timing race with `activate()`), and the click lands on the URL bar / a tab / a button when the title-bar drag area is too narrow (Firefox in particular). Behavior inconsistent per-window ‚Äî too fragile to ship.
      - **The yabai-style fix** is to install a scripting addition into the Dock (SIP-disabled) and route AX writes through it. Same dead-end as gotcha 13; out of scope for a launcher.

    Combined with gotcha 13, the consequence is that we can't reliably *list* a window in the launcher AND focus it on activation. After trying both narrower scopes (active Space only, then active Space + active display), the user could still pick a listed window and end up with the wrong one focused ‚Äî AND the listed set would silently change as the cursor moved between displays. Both are worse-than-no-feature. So the window switcher is disabled on macOS; window-action commands continue to work because they don't require the user to pick a specific window ‚Äî they always act on the **globally** frontmost non-LoFi window, which is reliably identifiable from CGWindowList's z-order. The "active-display filter" added during the switcher's lifetime was removed once the switcher was disabled: scoping the command target to the cursor's display caused a subtle bug for "Next display" / "Previous display" (the cursor stays put when you move a window between displays, so the next LoFi invocation would target a *different* window on the cursor's display rather than the foreground window the user moved). The investigation lives in this project's memory under `project_ax_cross_display_focus.md`.
15. **A borderless window needs a clear, non-opaque background for rounded corners and a shape-hugging shadow.** The Spotlight look is a rounded glass content view inside the window. If the window stays opaque with a solid `backgroundColor`, its own square corners paint *behind* the rounded content and the system shadow (drawn from the window's opaque silhouette) traces the square, not the rounded shape. `PanelController` sets `isOpaque = false`, `backgroundColor = .clear`, and `hasShadow = true` so the rounded `NSGlassEffectView` defines both the visible shape and the shadow outline.
16. **Liquid Glass (`NSGlassEffectView`) is macOS 26-only; the deployment target is 15.0.** The Tahoe glass surface must be behind `if #available(macOS 26.0, *)`, with an `NSVisualEffectView` (`.hudWindow`, `.behindWindow`, layer `cornerRadius` + `masksToBounds`) fallback for 15.x‚Äì25.x. `PanelController.makeBackground()` returns a `(container, host)` pair so the rest of the layout is backend-agnostic: `container` becomes the panel's `contentView`, `host` (the glass's `contentView`, or a pinned subview of the effect view) carries the search field + list. For the glass to actually show through, the search field (`drawsBackground = false`) **and** the results (`NSScrollView.drawsBackground = false`, `NSTableView.backgroundColor = .clear`) must be transparent ‚Äî otherwise only the search strip looks like glass.
17. **A borderless `NSSearchField` can't align its magnifier ‚Äî stripping the bezel makes the cell ignore `searchButtonRect`/`searchTextRect`.** `NSSearchFieldCell` positions the magnifier and text through those rect methods, but only while it draws its bezeled search appearance. With `isBezeled = false` (which the borderless look requires) the cell falls back to plain `NSTextFieldCell` drawing and ignores them, so the glyph and the typed text collide at the left edge ‚Äî overriding the cell (via `cellClass`) doesn't change that once the bezel is gone. The fix is to drop `NSSearchField` entirely and compose the input as `SearchHeaderView`: a magnifier `NSImageView` (a `magnifyingglass` SF Symbol in a `kIconSize` box) plus a borderless `NSTextField`, laid out with the *same* insets and spacing as `EntryRowView`. Sharing the row's `[icon] text` layout is what guarantees the magnifier and text line up with the app icons and names below. The controller's `controlTextDidChange` / `control(_:textView:doCommandBy:)` are `NSControlTextEditingDelegate` methods and work unchanged on the plain `NSTextField`; the panel pins the inner field (not the header container) as `initialFirstResponder` so the field editor activates on show. Two more things had to line up on the *row* side: (a) the table runs in `.plain` style, because the default `.automatic`/`.inset` style adds a hidden horizontal content inset (~15pt) that shoves every row right of the header; and (b) since `.plain` drops the inset style's rounded selection, `RoundedSelectionRowView` redraws that pill itself in `drawSelection(in:)`. Net: header and rows share one `kCellHorizontalPadding` column, and selection still looks like the system inset highlight. (The search input also uses a larger `kSearchFontSize` for a Spotlight-style prompt ‚Äî purely cosmetic, independent of alignment.)

### Window-action commands (AX + coordinates)

18. **Two coordinate spaces, flipped exactly once.** AX (`kAXPositionAttribute`/`kAXSizeAttribute`) and `kCGWindowBounds` are **top-left global** (origin at the primary display's top-left, y growing downward). `NSScreen.frame`/`visibleFrame` are **Cocoa bottom-left** (origin bottom-left, y up). The two have to agree before a rectangle round-trips through `compute_geometry` and back into AX. The work area is the *only* value that needs flipping (the target window's current frame already comes from `kCGWindowBounds`, top-left). `WindowCommands.workAreaTopLeft` does it with `topLeftY = primaryHeight - vf.origin.y - vf.height` where `primaryHeight = NSScreen.screens[0].frame.height` ‚Äî `vf.origin.y` is the distance from the primary display's *bottom* to the visible frame's bottom edge, so the top-left y is `primaryHeight - (vf.origin.y + vf.height)`; x is unchanged. The flip **must** use `screens[0]` (the menu-bar/primary screen, which defines the global origin for *both* spaces) even for a target window on a secondary monitor; using the target screen's own height mis-places the rect on multi-monitor setups. `compute_geometry` itself is origin-agnostic pure arithmetic, so a top-left work area in yields a top-left rect out that is usable directly as an AX position/size ‚Äî `WindowControl` never flips. Empty `NSScreen.screens` ‚áí nil work area ‚áí no command rows; an off-screen window falls back `NSScreen.main` ‚Üí `screens[0]`.
19. **Toggle-maximize is a *true toggle*, because macOS has no app-independent maximize.** GNOME uses Mutter `maximize()`/`unmaximize()`, and Mutter remembers the pre-maximize frame so un-maximize restores the exact previous size. AX exposes neither an app-independent maximize nor a saved frame, and LoFi is a short-lived process that quits after each activation ‚Äî so the maximize press (run 1) and the un-maximize press (run 2) happen in two different process lifetimes and in-memory state can't bridge them. `WindowControl.toggleMaximize` therefore persists the pre-maximize frame itself in `SavedFrameStore` (UserDefaults ‚Äî the idiomatic small-state store for a bundled `LSUIElement` app, no file I/O to get wrong), keyed by CGWindowID, with a save-on-maximize ‚Üí consume-on-restore lifecycle plus a startup `prune(liveWindowIds:)` that drops frames for windows no longer on screen. The divergence from GNOME is *only* in mechanism (we store the frame; Mutter does); the user-visible behavior matches. The discriminator is **geometry, not "does a saved frame exist"**: a window "fills" the work area iff all four edges are within `kMaximizeFillTolerance = 2`pt of it (AX-reported geometry is off by a point or two from window-server rounding, so an exact `==` would never register a maximized window). Not-filling ‚áí save the current frame, then resize to the work area; filling ‚áí restore the saved frame (read-and-removed), or the **StandardSize fallback rect** when none was saved (e.g. the window was already maximized before LoFi first saw it). Choosing geometry as the discriminator means a window the user manually shrinks after a LoFi-maximize is no longer "filling", so the next press re-maximizes it (overwriting the saved frame) ‚Äî the intuitive result. The StandardSize fallback is single-sourced from Rust's `compute_geometry` (not re-derived in Swift): after pushing the commands, `AppDelegate.pushCommands` scans the pushed rows **by id** for `standard_size` and reads its `commandGeometry(at:)`. The CGWindowID-reuse staleness risk (a closed window's id reassigned to a different window, which then restores to the wrong size) is bounded by the save‚Üíconsume lifecycle and the startup prune; a single benign wrong-size restore is the worst case.
20. **AX geometry only sticks with `AXEnhancedUserInterface` OFF, applied as `size ‚Üí position ‚Üí size`.** Two separate traps conspire here, and getting either wrong yields a window that *either* moves *or* resizes but not both:
    - **`AXEnhancedUserInterface` must be off during geometry.** `AXWindowFinder.windowsForApp` turns it *on* to wake sleeping AX runtimes for enumeration (gotcha 12), but with it on the window server applies position/size changes asynchronously and animated, so the two sets fight and only whichever was set **last** survives. Observed on Ghostty: `position‚Üísize` left it resized-but-not-moved; `position‚Üísize‚Üíposition` left it moved-but-not-resized ‚Äî the tell-tale "last write wins" fingerprint. `WindowControl.disableEnhancedUI` turns it back off immediately before any geometry set; we leave it off (off is the normal state and LoFi quits right after). Rectangle and yabai do the same.
    - **The order is `size ‚Üí position ‚Üí size`.** Even with the sets synchronous, a single pass gets clamped: the server constrains a *move* using the window's size at that instant (a too-wide window can't move left without its right edge running off-screen, so the move is silently clamped back ‚Äî the set still returns `.success`) and a *resize* against its position. Setting size first shrinks the window so the subsequent move isn't clamped; the trailing size re-asserts dimensions once the window is at its target origin (covering the grow case). `WindowControl.setFrame` does all three.

    Separately, AX silently drops geometry on a window that is currently *fullscreen*, so both `move` and `toggleMaximize` clear the undeclared `"AXFullScreen"` attribute first (bare CFString ‚Äî no `kAX` constant ‚Äî same pattern as the `"AXEnhancedUserInterface"` wakeup; read-back coerced via `(value as? Bool)` since it returns a CFBoolean), mirroring GNOME's MoveResizeWindow unmaximize/unfullscreen. There is also no public "is zoomed" attribute, so the explicit position+size set is what de-zooms a green-button-maximized window in practice. Some apps don't support `"AXFullScreen"` at all; the set then isn't `.success` and the fullscreen toggle returns false (best-effort). Finally, terminals (Ghostty) snap their size to the character-cell grid, so the achieved width/height can land a few points under the request ‚Äî expected quantization, not a bug.
21. **The command target is the frontmost *on-screen, current-Space, globally-z-ordered* non-LoFi window; no target ‚áí no command rows (GNOME parity).** The commands are gathered, like GNOME's `gather_commands`, against a single *target window*: the frontmost non-LoFi window. The set is fourteen commands on a single-display setup and sixteen on multi-display (Next display + Previous display are appended when `NSScreen.screens.count >= 2`). `WindowDiscovery.discover` returns the active Space's windows in reliable front-to-back z-order (`.optionOnScreenOnly`), so `WindowCommands.gatherTarget`'s `.first { non-LoFi }` is the window the user was just using regardless of which display it's on. `gatherTarget` returns nil when there's no non-LoFi window on the active Space at all, and `AppDelegate` then pushes zero command entries ‚Äî the rows simply don't appear, matching GNOME where an empty `gather_commands` drops the rows entirely. The gather is also belt-and-suspenders against the LoFi bundle id (`dev.jplein.lofi`) on top of `WindowDiscovery`'s `pid == getpid()` filter. `AppListController` tolerates a nil `commandTarget` (the `"Command"` launch branch becomes unreachable but bails safely if entered).

    **No active-display filter on the command target.** An earlier iteration filtered the target by the cursor's display, originally added for the window switcher (gotcha 14) and kept for commands as a "tidiness" measure. It was actively wrong for commands: after a "Previous display" / "Next display" move, the user's cursor stays on the originating display while the foreground window has moved to the destination display; on the next LoFi invocation, the cursor-filtered `gatherTarget` would pick whatever happens to be on the cursor's display (a *different* window) instead of the window the user just moved ‚Äî so "Next display" would then move the wrong window, and sometimes no non-LoFi window would be on the cursor's display at all, dropping the command rows entirely. The fix was to drop the filter: the command target is now the globally frontmost non-LoFi window, and each command's *work area* is still derived from the *target window's* display via `WindowCommands.workAreaTopLeft`, so geometry commands (Center, halves, etc.) keep the window on its own display.

### Hotkey + daemon lifecycle

22. **`RegisterEventHotKey` is the only API that can register a global hotkey AND consume the keystroke ‚Äî even though it lives in Carbon.** The system-wide Option+Space hotkey is implemented in `Hotkey.swift` against Carbon's `RegisterEventHotKey` / `InstallEventHandler` (from `HIToolbox`). The natural-looking AppKit alternative `NSEvent.addGlobalMonitorForEvents(matching: .keyDown, ...)` doesn't work for our case: it requires Accessibility permission, and crucially **cannot consume the event** ‚Äî Option+Space would still fall through to whatever app has focus (which is rarely what users want from a launcher hotkey). `CGEventTap` could consume but introduces latency on every keystroke. Carbon-the-framework is broadly deprecated, but `RegisterEventHotKey` specifically has no AppKit replacement, ships on Apple Silicon + macOS 26, and is what every shipping macOS launcher uses (Alfred, Raycast, Rectangle, Hammerspoon, BetterTouchTool); the popular Sindre Sorhus `KeyboardShortcuts` Swift package is a thin wrapper around it. We don't pull in the package because the call site is ~50 lines and adding a SwiftPM dependency to a Bazel build is friction we don't need.

    The C event handler callback can't capture `self` (it's a `@convention(c)` function pointer), so we pass `Unmanaged.passUnretained(self).toOpaque()` as the handler's `userData` and unwrap it back inside the trampoline. *Unretained* is correct because the `GlobalHotkey` instance is owned by `AppDelegate` for the entire process lifetime; `deinit` unregisters the handler before the pointer would dangle. If we ever move to per-screen or per-window hotkey lifetimes, this needs revisiting.
23. **`applicationDidFinishLaunching` does NOT show the panel; the first paint happens in the hotkey handler.** Before the daemon slice, `applicationDidFinishLaunching` discovered apps, gathered the command target, applied MRU, and called `panel.makeKeyAndOrderFront`. Now it does the *summon-stable* setup once (cache `[DiscoveredApp]`, open the SQLite MRU store, build the panel + list controller empty, register the hotkey) and returns without showing anything. The first visible paint happens when the user presses Option+Space (or `:activate` sends a Launch Services reopen event), which calls `AppDelegate.summonPanel` to: `entries.clear()` (drop the previous summon's contents), re-push the cached app set, gather the *current* frontmost-non-LoFi window + push the commands, apply MRU, hand the new command target to the list controller, then `NSApp.activate(...)` + `panel.show()`. (One exception: the first summon that finds a TCC permission missing fires the system prompts and returns *without* painting, so the `.floating` panel doesn't bury the permission dialogs ‚Äî see *Permissions* above.)

    The `entries.clear()` step is what `lofi_entries_clear` (Rust FFI, added with this slice) is for ‚Äî without it every summon would *append* a fresh copy of apps + commands to the existing list and the user would see duplicates plus stale command targets. The command-target gather happens *before* `NSApp.activate` so `WindowDiscovery.discover` reads `CGWindowListCopyWindowInfo` while LoFi is still background, which keeps the frontmost-non-LoFi z-order pointing at the user's previous app rather than at LoFi itself. (The `pid == getpid()` filter would catch LoFi anyway, but the z-order is more useful when LoFi isn't in the way.)
24. **`hidesOnDeactivate = true` couples key status and visibility, so `NSApp.isActive` is the right toggle discriminator.** `PanelController` sets `hidesOnDeactivate = true` so the panel auto-hides when LoFi resigns key (e.g. the user picks "Open Safari" and Safari activates). The hotkey handler (`AppDelegate.toggleOrSummon`) checks `NSApp.isActive` rather than `panel.isVisible` to decide between dismiss and summon: with `hidesOnDeactivate`, those two states are coupled tightly enough that they always agree, and `isActive` is the cleaner signal for "the user is currently interacting with LoFi." `applicationShouldHandleReopen` (the Launch Services hook used by `:activate`) returns `false` because LSUIElement apps have no default window ‚Äî returning true would have AppKit try to open one. Esc and after-activation paths go through `AppDelegate.dismissPanel` (`panel.orderOut` + `NSApp.hide`) rather than `NSApp.terminate`; Cmd-Q in the hidden Application menu is the only terminate path, and `:close` is the from-terminal equivalent.

### Bazel

25. **`DEVELOPER_DIR` set by the Nix devShell points at a partial Darwin SDK** in the nix store, which doesn't contain a usable Swift toolchain. `rules_swift` walks `xcrun --find swiftc` against `DEVELOPER_DIR` and bails. `.envrc` explicitly re-exports `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer` after `use flake` to override.
26. **`apple_support` must appear above `rules_cc` in `MODULE.bazel`.** Module ordering determines toolchain registration order; if `rules_cc` registers first, rules_swift picks up the generic CC toolchain (target triple "local") and fails. Reordering is non-obvious from the error message.
27. **`swift_interop_hint` auto-generates the Clang module map** ‚Äî do not also put a hand-written `module.modulemap` in `cc_library.hdrs`. It gets included as a C header in the auto-generated map and the parser barfs on module-map syntax.
28. **`crate_universe`'s binary targets are named `<crate>__<bin>`**, e.g. `@crates//:cbindgen__cbindgen`, not `__cli` or `__bin`. `gen_binaries` takes a list of binary names, not a `True` boolean.
29. **cbindgen 0.29 has no `--features` CLI flag** ‚Äî features are discovered by running `cargo metadata --all-features` internally. Passing `--features ffi` to the binary errors out; the right approach is to let cbindgen compute it.

## Out of scope this slice

Each is a follow-up:

- Workspace commands. Mutter-style workspaces have no direct macOS analog (Spaces are the closest, but they're not first-class targets the way Mutter workspaces are). This covers both the **switch-to-workspace** entries and the **move-window-to-workspace** commands ("Move to workspace N" / "Move to previous/next workspace"). The latter exist in `lofi-core` as `Entry::WorkspaceCommand` and are surfaced on GNOME (see `app/gnome/README.md`), but the macOS frontend deliberately never pushes them ‚Äî `lofi_entries_get_category` would report them as `"WorkspaceCommand"`, a string the macOS side never actually receives because nothing constructs the entries. (Window-action commands ‚Äî center/halves/two-thirds/standard-size/minimize/toggle-maximize/toggle-fullscreen ‚Äî and power commands ‚Äî Lock/Log Out/Sleep/Restart/Shutdown ‚Äî are now **implemented**; see the *Status* section, the *Power commands* section, and gotchas 18-21 above.)
- **The window switcher (listing open windows as launcher entries).** Implemented and shipped, then disabled. Two independent macOS Tahoe limitations make it unreliable for an unprivileged-process launcher: `SLSManagedDisplaySetCurrentSpace` from a non-Dock-injected process can't actually switch Spaces (gotcha 13), and AX writes that retarget another app's key window across a display boundary are silently dropped (gotcha 14). Narrowing the listing scope to (a) active Space, then (b) active Space + active display, made the list contents depend on mouse-cursor position AND still left the cross-display focus problem within the active display when the target app had sibling windows elsewhere ‚Äî picking a listed window could still fail to focus it. That's worse-than-no-feature. The only known path that *would* work is yabai-style Dock injection (SIP-disabled scripting addition routing private SkyLight calls through the Dock); that requirement is non-negotiable for the people who use it and is out of scope for a launcher. Users who want to reach a specific window should use the macOS-native flow: Cmd-Tab (with the "switch to a Space with open windows" preference on, in System Settings ‚Üí Desktop & Dock), Mission Control, or Dock-click. The window-action commands (center, halves, minimize, etc.) are unaffected ‚Äî they don't require the user to pick a specific window, they always act on the frontmost non-LoFi window on the active display, which IS reliably identifiable.
